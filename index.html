How the Engine Picks a Price

Every pricing query goes through these 8 steps. The engine logs the full trace of every step for audit purposes.

1

Accept the request

Receive: customer ID, product SKU, quantity, and pricing date. Assign a unique transaction ID for traceability.

2

Look up the product

Find the product and its full hierarchy: BU, MAG, AG, MG3, MPG. The engine needs these to search for discount rules at different levels.

3

Get the list price

Look up the country price list active on the pricing date. Example: pricing on Mar 15 2026 uses the US-2026 price list (valid Jan 1 - Dec 31 2026).

4

Find all contracts for this customer

Walk UP the customer hierarchy to collect contracts. A hospital inherits contracts from its parent IDN and grandparent GPO. See Section 4 for details.

5

Check product scope

For each contract, verify the product is in scope. A contract scoped to BU=Connected Care will not price a Diagnostic Imaging product. Contracts that don't cover this product are eliminated. See Section 5.

6

Find the discount (bottom-up search)

For each remaining contract, search for the most specific discount rule: SKU first, then MPG, then AG+MG3, all the way up to BU. First match wins. No compounding. See Section 6.

7

Calculate the net price

Apply the discount using the rule's type (Price Float, Fixed Price, etc.). Each type uses a different formula. See Section 7 Rule Cards for all formulas.

8

Pick the winner

If multiple contracts produced a price, select the winner using the resolution strategy: BEST_PRICE (lowest price wins) or HIERARCHY (closest contract wins). See Section 9. Log the full trace.

Customer Entitlement Chain

Customers are organized in a 3-level hierarchy: GPO > IDN > Hospital. When a hospital asks for a price, the engine walks UP the chain and collects contracts from every level. The hospital does not need its own contract -- it can inherit pricing from its parent IDN or grandparent GPO.

Walkthrough
Memorial Hospital (H-MEM) wants a price for MON-8500X. The engine walks up:
  • Step 1: Check H-MEM's own contracts. None found.
  • Step 2: H-MEM's parent is IDN HealthFirst. Check IDN-HF's contracts. Found: C-002.
  • Step 3: IDN-HF's parent is GPO Premier. Check GPO-PREM's contracts. Found: C-001.
  • Result: H-MEM is entitled to both C-001 and C-002. The engine prices MON-8500X under each contract and picks the better deal.

Note: Some hospitals have direct contracts too. For example, Central Surgical Center (H-CSC) has contract C-006 (Direct) AND inherits C-001 from GPO Premier. Independent hospitals like University Research Hospital (H-URH) only have their own Framework contract (C-003).

Contract Product Scope

A contract does not cover every product in the Philips catalog. Each contract defines which products it applies to using a "product scope" -- one or more entries that specify a part of the product hierarchy. If a product does not fall within the contract's scope, that contract is skipped during pricing.

Neither contract covers this product, so the engine returns the full list price with no discount.

Discount Hierarchy (Bottom-Up Search)

When the engine looks for a discount rule within a contract, it starts at the most specific level (the exact product SKU) and works upward to the broadest level (the entire Business Unit). It stops as soon as it finds a match. Discounts are never stacked or combined.

Example: Bottom-Up Search for MON-5500S under GPO Premier (C-001)
MON-5500S is in: BU=Connected Care, MAG=Monitoring Systems, AG=Patient Monitors, MG3=NEW, MPG=MPG-BEDSIDE-MON. C-001 has rules at SKU (R6 for MON-8500X only), AG (R1b for Patient Monitors at 24%), and BU (R1a for Connected Care at 18%).
  • SKU? Looking for MON-5500S. No rule. Continue.

Rule Cards

Price Float (Percentage) #1

Implemented
The discount percentage is applied to whatever the list price is today. If Philips raises the list price next month, the customer's net price goes up too (they pay more). If Philips lowers the list price, the customer's net price goes down (they pay less). The price "floats" with the market.
Formulanet = list(today) x (1 - d%)
Worked Example (using actual seed data)
Memorial Hospital orders MON-8500X (IntelliVue MX850). GPO Premier contract C-001 gives 28% off this specific SKU (rule R6). US-2026 list price: $18,500.
  • Mar 15 2026: List = $18,500 (US-2026). $18,500 x 0.72 = $13,320
If Philips published a US-2027 price list with MON-8500X at $19,500, the same query on Jan 2 2027 would yield $19,500 x 0.72 = $14,040. The customer pays $720 more just because the list price went up. If the 2027 price dropped to $17,000 instead, the customer pays $17,000 x 0.72 = $12,240, saving $1,080.
Key difference: The engine does not remember any previous prices. Every time someone asks "what does this cost?", it looks up today's list price and applies the discount fresh. The price can go up or down freely.

Price Float (Fixed Dollar) #1b

Implemented
Instead of taking a percentage off, a fixed dollar amount (e.g., $75) is subtracted from today's list price. Every product covered by this rule gets the same dollar discount, regardless of whether the product costs $5,000 or $50,000. Like Price Float, the price still moves when Philips changes the list price.
Formulanet = list(today) - $amount
net = MAX(0, net) // price can never go below zero
Worked Example (using actual seed data)
GPO National Alliance (C-004) has rule R-FD-1: a flat $75 off all Patient Monitors (AG level). This applies to every Patient Monitor SKU regardless of its price.
  • MON-8500X: List = $18,500. $18,500 - $75 = $18,425 (tiny 0.4% saving on an expensive product)
  • MON-4000B: List = $5,400. $5,400 - $75 = $5,325 (1.4% saving, same $75)
Key difference: This is the only variant that uses a dollar amount instead of a percentage. It can only be used with the Price Float discount type. You cannot combine fixed-dollar amounts with Fixed Price, Fixed or Lower, or Falling Ceiling rules.

Fixed Price #2

Implemented
The discount is applied to the list price that was in effect on the day the contract started, not today's price. The customer's net price is calculated once and stays the same for the entire contract, no matter what happens to list prices afterward. Philips raises prices? Customer still pays the original rate. Philips lowers prices? Customer still pays the original rate (they do NOT get the benefit of the drop).
Formulareference = list(contract start)
net = reference x (1 - d%)
Worked Example (using actual seed data)
Contract C-006 (Direct - Central Surgical Center) started Jul 1, 2025. On that date, the active price list was US-2025. MON-8500X had a list price of $17,575 in US-2025. Suppose C-006 had a Fixed Price rule at 15% for MON-8500X. The engine locks in $17,575 as the reference.
  • Jul 2025: Reference price locked. $17,575 x 0.85 = $14,938.75
  • Mar 2026: US-2026 list price is now $18,500. Doesn't matter. Still $17,575 x 0.85 = $14,938.75 customer is protected from the $925 increase
  • Hypothetical: If Philips dropped the list to $16,000, customer still pays $14,938.75 customer misses the drop
Key difference: The engine has to look up a historical list price (what was the price on the contract start date?). If that historical data is missing, it falls back to today's price and adds a warning to the trace. Unlike Price Float, this price never changes during the contract.

Fixed or Lower #3

Implemented
The engine calculates two prices every time: (a) the fixed price based on the contract-start list price, and (b) the floating price based on today's list price. The customer always pays whichever is lower. If Philips raises list prices, the customer is protected (the fixed price kicks in as a ceiling). If Philips lowers list prices, the customer benefits (the floating price drops below the fixed price). Best of both worlds for the customer.
Formulafixed = list(contract start) x (1 - d%)
float = list(today) x (1 - d%)
net = MIN(fixed, float) // customer always gets the cheaper one
Worked Example (using actual seed data)
Central Surgical Center (H-CSC) has contract C-006 (Direct, started Jul 1 2025). Rule R-FOL-2: 15% off MON-8500X as Fixed or Lower. The US-2025 list price for MON-8500X was $17,575. So the fixed ceiling = $17,575 x 0.85 = $14,938.75.
  • Fixed ceiling: $17,575 x 0.85 = $14,938.75 (this never changes during the contract)
  • Mar 2026: US-2026 list = $18,500. Float = $18,500 x 0.85 = $15,725. Compare: $14,938.75 vs $15,725 = $14,938.75 fixed is cheaper, customer protected from increase
  • Hypothetical: If list dropped to $16,000. Float = $16,000 x 0.85 = $13,600. Compare: $14,938.75 vs $13,600 = $13,600.00 float is cheaper, customer benefits from drop
  • Later: If list goes back to $19,000. Float = $19,000 x 0.85 = $16,150. Compare: $14,938.75 vs $16,150 = $14,938.75 back to the fixed ceiling
Key difference from Falling Ceiling: When list prices rise back up, the price CAN go back up to the fixed ceiling ($14,938.75 in this example). The engine does not remember the $13,600 from the hypothetical drop. Each query is calculated fresh with no memory of past prices.

Falling Ceiling #4

Implemented
The customer's price can only ever go down, never back up. When the first pricing query is made, the engine calculates the initial net price and saves it as the "watermark" (the lowest price so far). Every time a pricing query comes in, the engine compares today's calculated price against the watermark. If today's price is lower, it becomes the new watermark. If today's price is higher, the customer still pays the old watermark. Once the price drops, it stays down permanently for the rest of the contract.
Formulainitial_ceiling = list(contract start) x (1 - d%)
float = list(today) x (1 - d%)
net = MIN(watermark, float)
IF net < watermark THEN watermark = net // save the new low as the permanent ceiling
Worked Example (using actual seed data)
GPO Premier (C-001) has rule R-FC-1: 12% off VENT-7000 (Trilogy Evo ventilator) as Falling Ceiling. Contract started Jan 1 2026. The US-2026 list price for VENT-7000 is $24,500. Initial ceiling = $24,500 x 0.88 = $21,560.
  • First query: Watermark initialized. $24,500 x 0.88 = $21,560 this is saved in the database
  • Hypothetical: List rises to $26,000. Float = $26,000 x 0.88 = $22,880. Compare to watermark: $21,560 = $21,560 float is higher, customer pays the saved watermark
  • Hypothetical: List drops to $22,000. Float = $22,000 x 0.88 = $19,360. Compare to watermark: $21,560 = $19,360 float is lower! New watermark saved.
  • Hypothetical: List back to $28,000. Float = $28,000 x 0.88 = $24,640. Compare to watermark: $19,360 = $19,360 customer still pays $19,360, the drop is locked in forever
Key difference from Fixed or Lower: This is the ONLY rule where the engine remembers past pricing queries. It stores the lowest price ever given in a database table (falling_ceiling_watermarks). With Fixed or Lower, if the price drops to $19,360 and later rises, the customer goes back up to the fixed ceiling. With Falling Ceiling, the $19,360 is permanent. This is the most favorable rule for customers and the most expensive for Philips.
Open question for Philips: The watermark is NOT created when the contract starts. It is created the first time someone actually prices that product under the contract. The current implementation takes the lower of (a) the contract-start ceiling and (b) today's float price at the time of that first query. This means if list prices dropped between contract start and the first order, the watermark starts at the lower value, giving the customer a better starting point than the spec intended. Example: contract starts Jan 1 at $24,500 (ceiling = $21,560), but nobody orders until June when the list dropped to $22,000 (float = $19,360). The watermark initializes at $19,360, not $21,560. Is this the intended behavior, or should the watermark always initialize at the contract-start ceiling regardless of when the first query happens?
Implementation note: Simulations and pricelist generation do NOT update the watermark. This is by design: running a "what-if" scenario or generating a report should never permanently change a customer's price. Only actual pricing transactions (quotes and orders) can ratchet the watermark down.

Bundle & Option Discounts #5

Implemented
A bundle is a product package (e.g., an ICU Room Monitoring Package) that contains a core product plus optional add-ons. The bundle itself gets a contract discount just like any standalone product. The add-on options get their own separate discounts, which can be a flat dollar amount or a percentage. Important: the same option (e.g., a 512GB storage upgrade) can have a different discount when purchased inside a bundle vs. purchased on its own.
Bundle Formulabundle_net = bundle_list_price x (1 - bundle_d%)
option_discount = fixed_amount OR (option_list_price x option_d%)
total = bundle_net - sum(option_discounts)

How option discounts are found1. First, look for a discount specific to this option + this bundle
2. If none found, fall back to the generic discount for this option
Worked Example (using actual seed data)
A hospital orders ICU-ROOM-PKG bundle (list $45,000) under GPO Premier (C-001). Rule R-BUNDLE-1 gives 25% off the bundle. They add a 512GB storage module (OPT-MEM-512, standalone list $800). This option has two discount entries: $75 off when standalone (OD-001) and $60 off when inside the ICU bundle specifically (OD-002). Since it's inside the ICU bundle, OD-002 applies.
  • Bundle price: $45,000 x 0.75 = $33,750
  • Option discount: 512GB storage gets the bundle-specific $60 off (OD-002), not the standalone $75 off = $60 off
  • Total: $33,750 - $60 = $33,690
Key difference: Options inside a bundle show $0 as their list price (their cost is included in the bundle price). The option discount is an additional reduction. If the same option is bought standalone, it gets a normal hierarchy discount and the standalone option discount. Also note: bundle pricing always uses the BEST_PRICE resolution strategy, even if HIERARCHY is set for standalone products.

Contract Resolution Strategies

A hospital can be entitled to multiple contracts at the same time (e.g., their parent IDN's contract AND their grandparent GPO's contract). When this happens, the engine needs to decide which contract to use. There are two strategies.

BEST_PRICE

Default
Calculate the price from every contract the customer is entitled to, then give them whichever contract produces the cheapest price.
Logicwinner = contract with MIN(net_price)

HIERARCHY

Toggleable
Pick the contract based on which organization is closest to the customer. A hospital's own direct contract beats its parent IDN's contract, which beats its grandparent GPO's contract. If two contracts are the same type, the cheaper one wins.
Priority Order1. Framework/Direct (closest)
2. IDN
3. GPO (broadest)
Example: Memorial Hospital (H-MEM) ordering MON-8500X ($18,500)
H-MEM is entitled to C-001 (GPO Premier) and C-002 (IDN HealthFirst). Under C-001, the bottom-up search finds R6 (SKU-level, 28%). Under C-002, the search finds R2a (MAG-level, 22%). Note: R6 only exists on C-001, so C-002 doesn't get the SKU-level match.
  • C-001 (GPO): $18,500 x 0.72 = $13,320
  • C-002 (IDN): $18,500 x 0.78 = $14,430
  • BEST_PRICE: $13,320 via GPO Premier (GPO contract wins because it gives the lowest price)
  • HIERARCHY: $14,430 via IDN HealthFirst (IDN is closer to the hospital than GPO, so it wins even though the price is higher)

Contract Version Drafting

When contract terms need to change (e.g., a discount goes from 18% to 22%), the engine supports a draft-then-activate workflow so changes can be reviewed before going live. Rules have a 3-state lifecycle:

Draft
Invisible to pricing engine
→
Active
Used by pricing engine
→
Expired
Replaced by new version
StepWhat happensImpact on live pricing
1. DraftNew rules are created with status = "draft". Multiple rule changes can be grouped into a single draft version.None. Draft rules are invisible to the pricing engine.
2. CompareThe engine simulates all draft rules and shows before/after pricing impact across all affected hospitals and SKUs.None. This is a read-only simulation.
3. ActivateAn atomic transaction: old matching rules get expired (valid_to set to activation date - 1 day, status = "expired"), draft rules become active (valid_from = activation date, status = "active"), and the contract version number increments.Immediate. New prices take effect from the activation date.
Orphaned drafts (created but never activated) remain in the database but have no effect on anything. Every rule mutation (create, update, delete, activate) is logged in the rule_audit_log table with the original natural language input for audit trail.
